## 0. 开始前准备

### 需要安装的软件

1. 安装 [Visual Studio Code](https://code.visualstudio.com/)。安装时可以使用默认选项。
2. 安装 Git。macOS 可以在终端执行 `git --version`；如果系统提示安装 Xcode Command Line Tools，按提示安装即可。
3. 准备一个 GitHub 账号，并确认自己有 `Hyakimaru7/personal-homepage` 仓库的写入权限。如果这是你自己的仓库，登录对应账号即可。

如果不确定 Git 是否安装成功，在 macOS 的“终端”中执行：

```bash
git --version
```

看到类似 `git version 2.x.x` 的文字就可以继续。VS Code 和 Git 是两个不同的软件：VS Code 用来编辑文件，Git 用来记录修改并上传到 GitHub。

### 项目文件分别做什么

| 文件或文件夹 | 用途 | 新手通常是否需要修改 |
| --- | --- | --- |
| `index.html` | 页面文字、标题、链接和板块结构 | 是，修改内容主要在这里 |
| `styles.css` | 颜色、字体、间距、布局和手机端样式 | 需要改样式时再改 |
| `script.js` | 菜单、滚动动画、笔记筛选和页脚年份 | 一般不改 |
| `assets/portrait.jpg` | 首页个人照片 | 可替换 |
| `assets/research-space.png` | 研究板块配图 | 可替换 |
| `assets/notes/covers/` | 课程笔记封面图 | 可替换或新增 |
| `assets/notes/` | 笔记 PDF 文件 | 通过 GitHub Releases 管理 |
| `scripts/deploy-github.sh` | 首次配置仓库和 Pages 的自动脚本 | 一般不改 |

不要把整个项目文件夹复制成另一层同名文件夹。例如正确的文件路径是 `personal-homepage/index.html`，而不是 `personal-homepage/personal-homepage/index.html`。

### 在 VS Code 中修改的标准流程

每次修改都可以按下面的固定流程进行：

1. 在左侧资源管理器中点开要修改的文件。
2. 用 `Command + F` 搜索页面上已有的文字或标签，例如搜索 `罗凯锋`、`section id="about"`。
3. 只修改需要替换的文字、数字、文件名或链接，不要误删成对的 HTML 标签。
4. 按 `Command + S` 保存。文件标签上的白点消失，表示已经保存。
5. 在浏览器中刷新本地页面，检查电脑端和手机端效果。
6. 确认无误后，再进行 Git 提交和上传（见第 13 节）。

VS Code 左侧文件名旁的 `M` 表示文件已修改，`U` 表示新文件尚未被 Git 跟踪。这些标记是正常的，不是报错。修改过程中可以使用 `Command + Z` 撤销，使用 `Command + Shift + Z` 重做；不要为了“清除标记”而随意删除文件。

### HTML 修改时最重要的规则

- HTML 标签通常成对出现，例如 `<p>文字</p>`。修改内容时保留标签本身，只替换中间的文字。
- 链接写在 `href="..."` 中，图片写在 `src="..."` 中；引号必须是英文半角双引号。
- `id` 用于导航定位，导航中的 `href="#about"` 必须对应某个元素的 `id="about"`，两边拼写要完全一样。
- 文件名和文件夹名区分大小写。`Portrait.jpg`、`portrait.jpg` 和 `portrait.JPG` 可能被当成不同文件。
- 不要在文件名中使用空格、中文括号或 `/`。图片和 PDF 推荐使用英文小写、数字和连字符，例如 `topology-notes.pdf`。
- 如果要在文字中显示 `<` 或 `>`，不要直接当作普通文字写入 HTML；不确定时先保留原结构。

### 不用安装插件也能预览

本项目是原生 HTML、CSS 和 JavaScript，不需要 `npm install`，也不需要构建命令。使用上一节的 `python3 -m http.server 4173` 就能预览。若端口 `4173` 已被占用，可换成 `4174`，然后访问 `http://127.0.0.1:4174`。

## 7. 课程笔记

在 `index.html` 中查找：

```html

```

笔记板块开头可以修改：

- 笔记介绍。
- 笔记总册数。

每份笔记对应一个完整的卡片：

修改现有笔记时，需要检查以下内容：

- 封面链接中的 PDF `href`。
- “阅读笔记”按钮中的 PDF `href`。
- 封面图片 `src`。
- 课程分类。
- 页数。
- 笔记标题。
- `aria-label` 和 `alt` 说明。

笔记分类对应关系：

| 属性值 | 页面分类 |
| --- | --- |
| `analysis` | 数学分析 |
| `algebra` | 代数 |
| `equations` | 方程 |

`data-note-category` 必须和筛选按钮的 `data-note-filter` 保持一致。

### 替换现有笔记 PDF

把新 PDF 放入 `assets/notes`，并保持原来的英文文件名，然后运行：

```bash
cd /Users/luojunhui/Documents/personal-homepage

gh release upload notes-v1 \
  assets/notes/mathematical-analysis-1.pdf \
  --clobber \
  --repo Hyakimaru7/personal-homepage
```

把命令中的文件名替换成实际要更新的笔记文件名。文件名不变时，网页链接不需要修改。

### 新增一份笔记

1. 把 PDF 放入 `assets/notes`，建议使用英文小写文件名，例如 `topology-notes.pdf`。
2. 把封面放入 `assets/notes/covers`。
3. 在 `index.html` 中复制一个完整的 `note-card`。
4. 修改卡片标题、页数、分类、封面和两个下载链接。
5. 更新笔记总册数和总页数。
6. 上传新 PDF：

```bash
gh release upload notes-v1 \
  assets/notes/topology-notes.pdf \
  --repo Hyakimaru7/personal-homepage
```

网页中的下载链接写成：

```text
https://github.com/Hyakimaru7/personal-homepage/releases/latest/download/topology-notes.pdf
```

### 不使用命令行上传 PDF

如果电脑没有安装 GitHub CLI，可以用浏览器上传：

1. 登录 GitHub，打开仓库的 Releases 页面：<https://github.com/Hyakimaru7/personal-homepage/releases>。
2. 打开标签为 `notes-v1` 的 Release，点击右上角“编辑”（铅笔图标）。如果还没有这个 Release，点击“Draft a new release”，标签填写 `notes-v1`，再创建 Release。
3. 在附件区域把 PDF 拖进去，文件名必须和 HTML 链接末尾的文件名完全一致。
4. 替换同名文件时，先删除旧附件，再拖入新文件，最后点击“Update release”。
5. 回到 VS Code，确认 HTML 中的地址仍然是 `https://github.com/Hyakimaru7/personal-homepage/releases/latest/download/文件名.pdf`，然后只提交 HTML 和封面图片的修改。

PDF 本身被 `.gitignore` 排除，不应通过普通的 `git add` 上传。GitHub 普通仓库对单个大文件也有大小限制，使用 Release 保存课程笔记更合适。Release 附件上传完成后，链接可能需要几十秒才恢复可用。

## 13. 在 VS Code 中保存并上传到 GitHub Pages

网页公开发布需要经过三步：

```text
在电脑上修改文件 → Git 提交（commit）→ 推送到 GitHub（push）
```

`commit` 只是把修改记录在本地，`push` 才会把记录上传到 GitHub。只保存文件或只点击 VS Code 的“提交”按钮，都不会更新线上网页。

### 13.1 第一次检查 Git 身份和远程仓库

在 VS Code 中打开“终端 → 新建终端”，执行：

```bash
cd /Users/luojunhui/Documents/personal-homepage
git remote -v
git branch --show-current
```

正常情况下会看到远程地址 `https://github.com/Hyakimaru7/personal-homepage.git`，当前分支是 `main`。如果没有显示远程地址，可以执行：

```bash
git remote add origin https://github.com/你的用户名/personal-homepage.git
git branch -M main
```

请把 `你的用户名` 换成实际 GitHub 用户名。不要把仓库地址改成个人主页的网址（Pages 地址和仓库地址不是一回事）。

第一次推送时，GitHub 可能要求登录。推荐安装 GitHub CLI 后执行 `gh auth login`，按提示选择 `GitHub.com`、`HTTPS` 和浏览器登录。也可以在 VS Code 的“源代码管理”面板中按推送按钮，按弹出的浏览器提示登录。不要把密码或 Personal Access Token 写进文件或发给别人。

第一次提交如果提示 `Please tell me who you are`，需要设置提交者姓名和邮箱。建议使用 GitHub 账号中已验证的邮箱，或 GitHub 提供的 `noreply` 邮箱：

```bash
git config --global user.name "你的 GitHub 用户名"
git config --global user.email "你的 GitHub 邮箱"
```

引号中的内容必须替换成自己的信息。这只设置 Git 提交署名，不会替你登录 GitHub。

### 13.2 方法 A：使用 VS Code 图形界面（推荐新手）

1. 修改文件并按 `Command + S` 保存。
2. 点击 VS Code 左侧的“源代码管理”图标，查看“更改”列表。
3. 点开每个文件检查差异，确认没有误删 HTML 标签后，点击需要上传的文件旁的 `+` 暂存。当前这份 `个人主页修改指南.md` 是本地说明文件；如果不希望它公开出现在仓库中，就不要暂存它。
4. 在顶部消息框写提交说明，例如 `更新个人简介和研究信息`，然后点击“提交”。
5. 点击“同步更改”或“推送”（向上的箭头）。如果询问是否先拉取远程更新，先拉取，解决冲突后再推送。
6. 打开仓库页面 <https://github.com/Hyakimaru7/personal-homepage>，确认最新提交出现在文件列表中。

如果看不到“推送”按钮，点击左下角的分支名称确认当前分支是 `main`，再在“源代码管理”的 `...` 菜单中选择“推送”。提示没有上游分支时，在终端执行一次 `git push -u origin main`。

### 13.3 方法 B：使用终端命令

每次发布普通文字、图片、样式或论文时，在项目目录执行：

```bash
cd /Users/luojunhui/Documents/personal-homepage
git status
git add index.html styles.css script.js assets
git diff --cached --stat
git diff --cached --check
git commit -m "更新个人主页内容"
git pull --rebase origin main
git push origin main
```

这里的顺序很重要：`status` 查看状态，`add` 暂存网页相关文件，`commit` 创建本地版本，`pull --rebase` 把本地提交接到最新线上版本之后，`push` 上传。若还修改了其他确实要公开的文件，可再执行 `git add 文件名`。若 `git commit` 提示 `nothing to commit`，说明没有新的已保存修改。如果 `git push` 成功，通常会看到 `main -> main`。

项目的 `.gitignore` 会忽略 `assets/notes/*.pdf`、个人地址文件和简历等文件，因此 `git add -A` 不会自动把这些内容上传到 Git 历史。不要使用 `git add -f` 强行加入不确定能否公开的文件。

推送后等待 GitHub Pages 构建完成，再打开 <https://hyakimaru7.github.io/personal-homepage/>；浏览器没有变化时使用 `Command + Shift + R` 强制刷新。

然后用 VS Code 打开这个目录，修改、预览并按上面的流程提交。不要在已有项目中再次执行 `git init`，否则容易造成嵌套仓库和推送错误。

### 13.5 在哪里确认发布成功

1. 在仓库的 Actions 页面查看 Pages 构建任务：<https://github.com/Hyakimaru7/personal-homepage/actions>。
2. 在仓库中选择“Settings → Pages”，确认使用 `Deploy from a branch`、分支 `main` 和目录 `/ (root)`。
3. 构建任务成功后再访问公开主页。第一次配置 Pages 或更新大文件时可能需要几分钟。

日常内容更新不需要重新运行 `scripts/deploy-github.sh`。这个脚本只用于首次配置仓库、GitHub Pages 和课程笔记 Release；普通修改直接使用 VS Code 的推送功能或 `git push origin main`。

## 14. 常见问题

### 修改后刷新页面没有变化

尝试强制刷新：

- macOS Chrome/Safari：`Command + Shift + R`
- 或关闭页面后重新打开。

### 图片无法显示

检查：

- 文件是否确实位于 `assets` 中。
- HTML 中的文件名是否完全一致。
- 英文大小写是否一致。
- 文件扩展名是 `.jpg`、`.png` 还是 `.webp`。

### PDF 链接打不开

检查网页链接中的英文文件名是否与 GitHub Release 附件名称完全一致，并进入以下页面确认附件已经上传：

<https://github.com/Hyakimaru7/personal-homepage/releases/tag/notes-v1>

### `git push` 后主页没有立即更新

GitHub Pages 发布需要一点处理时间。可以在仓库的 Actions 页面查看发布状态：

<https://github.com/Hyakimaru7/personal-homepage/actions>

### `git push` 提示 rejected 或 non-fast-forward

这表示 GitHub 上有本地没有的更新。先保存自己的文件，然后执行：

```bash
git pull --rebase origin main
```

如果没有冲突，再执行 `git push origin main`。不要使用 `git push --force`，强制推送可能覆盖别人已经上传的内容。

### 出现 merge conflict（合并冲突）

VS Code 会在冲突文件中显示 `Current Change`（本地内容）和 `Incoming Change`（GitHub 内容）。逐段选择正确版本，删除文件中残留的 `<<<<<<<`、`=======`、`>>>>>>>` 标记，保存后执行：

```bash
git add 冲突文件名
git rebase --continue
git push origin main
```

如果看不懂冲突，可以先执行 `git rebase --abort` 回到拉取前的状态，再备份自己的改动并寻求帮助。不要直接删除整个文件。

### 提示 `Authentication failed`、`Permission denied` 或无法登录

确认浏览器登录的是拥有该仓库权限的 GitHub 账号，并检查远程地址：

```bash
git remote -v
```

如果地址属于别人的账号，不能直接推送；应先让仓库所有者授予权限，或在自己的账号中 Fork 后修改远程地址。推荐使用 `gh auth login` 或 VS Code 的 GitHub 登录，不要把账号密码写进命令或脚本。

### 提示 `src refspec main does not match any`

通常表示本地还没有提交，或当前分支名字不是 `main`。先执行：

```bash
git branch --show-current
git status
```

如果项目确实还没有提交，先 `git add -A` 和 `git commit -m "首次提交个人主页"`；如果分支显示为 `master`，可执行 `git branch -M main` 后再推送。

### 图片或 PDF 在本地能打开，上传后打不开

线上环境使用的是 GitHub 仓库中的路径和文件名。检查相对路径是否以项目根目录为起点，例如 `assets/portrait.jpg`，不要写成电脑上的绝对路径 `/Users/luojunhui/...`。同时检查大小写、扩展名和 Release 附件名称是否完全一致。

### 不小心把密码、地址或其他私密资料提交了

立即停止推送，不要只在最新提交中删除文件，因为旧提交可能仍然保留。撤销公开内容、修改已经泄露的密码或令牌，并尽快联系仓库管理员处理 Git 历史。个人地址文件和简历已经在本项目 `.gitignore` 中排除，但仍应在提交前用 `git diff --cached` 检查。

### 想撤销刚才的本地修改

还没有提交时，可以在 VS Code 的文件差异页面选择“放弃更改”，或使用：

```bash
git restore 文件名
```

这会永久丢弃该文件未提交的修改，执行前请确认不再需要这些内容。已经提交但尚未推送时，优先使用 VS Code 的历史或 `git revert`，不要随意改写公共分支历史。

## 15. 一次完整更新示例

例如要把首页简介中的“数学建模”改为“优化与数学建模”，并替换个人照片：

1. 用 VS Code 打开项目，搜索旧文字并替换。
2. 把新照片命名为 `portrait.jpg`，放入 `assets`，确认 HTML 中仍写着 `assets/portrait.jpg`。
3. 启动 `python3 -m http.server 4173`，在浏览器检查首页、手机宽度和图片裁切。
4. 在 VS Code “源代码管理”中查看 `index.html` 和 `assets/portrait.jpg` 的差异。
5. 写提交说明 `更新首页简介和个人照片`，点击“提交”，再点击“同步更改”。
6. 打开 GitHub 仓库确认提交，再等待 Actions 完成。
7. 访问 <https://hyakimaru7.github.io/personal-homepage/>，强制刷新并点击导航、邮箱和论文链接。

完成这套流程后，下一次只需重复“修改 → 本地预览 → 提交 → 推送 → 检查线上页面”即可。
